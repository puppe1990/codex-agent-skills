# Ontology and Naming Checklists

Use these checklists when the initial review needs more rigor.

## 1. Ontology mapping

- What are the core entities?
- What are the core actions?
- Which concepts are primary versus secondary?
- Which terms are user-facing versus internal?
- Which concepts are currently grouped together?
- Which terms are near-synonyms but not identical?

## 2. User-language fit

- Would the target persona already know this term?
- Would they type this term into search?
- Would they expect to find this concept under this menu?
- Is this term native to the domain, or invented by the team?
- Does this term sound like a user task or a system capability?

## 3. Naming failures

- opaque codename
- overloaded term
- internal jargon
- false precision
- misleading generic term
- inconsistent action verb
- category name that hides the real task
- one concept, many labels
- many concepts, one label

## 4. Specificity versus consistency

Ask:
- Should this term match a broader system convention?
- Or should it become more specific at the point of user choice?
- Does consistency help transfer knowledge here?
- Does specificity improve recognition here?

## 5. Search and discovery

- Would this term work in app search?
- Would it work in docs search?
- Would a teammate find it in code search?
- Are there synonyms worth preserving?
- Does a rename need aliases, breadcrumbs, or cross-links?

## 6. Rename rollout

- which surfaces need to change together?
- do docs, APIs, and UI need different treatments?
- is backward-compatible language needed for a transition period?
- what old term should remain as an alias?
- what metric would show the rename helped?
